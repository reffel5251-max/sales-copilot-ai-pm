## 一、解决的业务问题

一线销售顾问的信息来源：政策群消息、前辈口传、自己整理的笔记、零散文档
知识获取成本高，导致三个核心问题：

| 问题 | 表现 | 影响 |
|------|------|------|
| 记不住，也查不快 | 权益/政策/金融方案分散在十几个群和备忘录 | 客户问了答不上来 |
| 说错了自己都不知道 | 政策更新靠群消息同步，可能用旧政策接待 | 口径不一致，信任受损 |
| 客户在等，自己在翻 | 算落地价要切换三个系统 | 接待节奏断掉，影响成交 |
| 客户提出质疑 | 话术靠老员工口传，新人上手慢，应对质量参差不齐 | 新人转化率低 |

## 二、产品设计思路

**定位**：做一个旁边站着的助手——输入客户情况，给推荐、给计算、给话术

### 六类核心处理链路

| 场景 | 实现方式 | 关键设计决策 |
|------|----------|--------------|
| 边界安全拦截 | 越界请求固定文本拒绝 | BOUNDARY 分支不走 LLM，防 Prompt 注入 |
| 模糊意图识别 | 固定文本回复引导重新对话 | LLM意图识别置信度 <0.75 拒绝作答，不强行生成 |
| 政策查询 | RAG 检索知识库 | 置信度 <0.75 拒绝作答，不强行生成 |
| 权益推荐 | RAG 检索知识库 + 库存查询API |接入库存查询AP，防止推荐无货 |
| 落地价计算 | 调用价格 API | 价格走 API 不走 LLM，杜绝数字幻觉 |
| 异议处理话术 | 话术库 RAG + LLM 生成 | 输出 2-3 个方向，不只给一个答案 |


### 三个关键产品判断

**价格走 API 不走 LLM**
数字的确定性比生成的灵活性更重要。价格幻觉是信任崩塌最快的方式，所以价格/库存类数据强制走工具调用，不允许 LLM 估算。

**越界请求硬编码拦截**
BOUNDARY 分支用固定文本节点输出，不走 LLM。任何 Prompt 注入都无法影响固定文本的输出，比 System Prompt 约束更可靠。

**知识库按主题分 7 个独立库**
单库全量召回会让无关内容占据 Top-3 名额。按意图优先搜对应库，Precision 显著提升。

## 三、系统架构

```
用户输入
  → LLM 意图识别（6类：POLICY/PRICE/RECOMMEND/SCRIPT/BOUNDARY/UNCLEAR）
  → 代码节点：槽位提取 + 置信度过滤（唯一判断点）
  → 变量赋值：更新会话上下文
  → 条件分支路由
      ├─ BOUNDARY  → 固定文本拒绝（不走LLM）
      ├─ UNCLEAR   → 澄清引导（不走LLM）
      ├─ PRICE     → 价格API → LLM生成明细
      ├─ RECOMMEND → 权益RAG + 库存API → LLM推荐
      ├─ SCRIPT    → 话术库RAG → LLM话术生成
      └─ POLICY    → 政策RAG → LLM摘要+来源标注
  → 变量聚合 → 输出
```

## 四、知识库设计

7个主题分库，按更新频率和语义边界划分：

```
knowledge_base/
├── 01_车型参数/        # 更新频率：季度
├── 02_权益政策/        # 更新频率：周级（高频，单独隔离）
├── 03_金融方案/        # 更新频率：月级
├── 04_上牌与购置税/    # 更新频率：月级
├── 05_异议话术/        # 更新频率：低，内容积累型
├── 06_交付与售后/      # 更新频率：月级
└── 07_常见问题FAQ/     # 持续积累
```

检索策略：混合检索（BM25 + 向量）→ Rerank → Top-3 → 置信度过滤

## 六、项目复盘

**意图识别最初只有5类，模糊输入无处归**
上线测试发现大量输入无法清晰归类，被强行分入错误分支。
→ 加入 UNCLEAR 第6类 + confidence 阈值过滤，代码节点作为唯一判断点。

**知识库最初是单文件，RAG 召回质量差**
所有政策放一个文件，无关内容占据 Top-3 名额，LLM 摘要质量差。
→ 改为7个主题分库，按意图优先搜对应库，召回精度显著提升。

**LLM意图分类置信度判断分散在多个节点**
条件分支和代码节点都在做 confidence 判断，出 bug 不知道去哪里找。
→ 判断逻辑全部收敛到代码节点，条件分支只做路由，职责单一。
